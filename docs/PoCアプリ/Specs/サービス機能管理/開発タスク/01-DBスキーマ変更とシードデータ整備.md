# 01 — DBスキーマ変更とシードデータ整備

## 概要

`services` コンテナのパーティションキーを `/id` から `/partitionKey` に変更し、ServiceFeature マスターデータ（19件）と TenantServiceFeature サンプルデータをシードスクリプトに追加する。

## 背景・目的

ServiceFeature ドキュメントを既存の `services` コンテナに `type: "service_feature"` として同居させるため、パーティションキーを `/partitionKey` に統一する必要がある。Azure Cosmos DB は既存コンテナのパーティションキー変更をサポートしないため、コンテナ定義の再作成が必要。同時にシードデータへの ServiceFeature / TenantServiceFeature データ追加を行い、開発環境の初期化で全データが自動投入されるようにする。

## 対象リポジトリ

- `scripts/` — DB作成スクリプトおよびシードデータの変更（親リポジトリ）

## タスク詳細

### 実装内容

1. **`scripts/create_database.py` の修正**
   - `create_service_database()` 内の `services` コンテナ作成処理のパーティションキーを `/id` → `/partitionKey` に変更する
   - `tenant_services` コンテナは変更なし（`/tenantId` のまま）

2. **`scripts/seed_data/initial_data.py` の修正**
   - 既存の `SERVICES` リスト内全 7 件の `Service` ドキュメントに `"partitionKey": "<id値>"` フィールドが追加済みであることを確認する（既存コードで確認済み — 既に `partitionKey` フィールドが存在する）
   - `SERVICE_FEATURES` リスト（19件）を追加する。データ定義は仕様書 02-システム要件仕様.md セクション 1.1 FR-02 の機能定義詳細に従う
   - 各項目: `id`, `type`, `service_id`, `feature_key`, `feature_name`, `description`, `default_enabled`, `partitionKey`（= `service_id`）, `created_at`

3. **`scripts/seed_data/sample_data.py` の修正**
   - `SAMPLE_TENANT_SERVICE_FEATURES` リストを追加する
   - サンプルテナントの割り当て済みサービスに対し、各テナント 2〜3 件のカスタム設定（`default_enabled` を上書きするデータ）を含める
   - フィールド: `id`（`{tenant_id}_{feature_id}`）, `type: "tenant_service_feature"`, `tenant_id`, `service_id`, `feature_id`, `feature_key`, `is_enabled`, `updated_at`, `updated_by`, `partitionKey`（= `tenant_id`）

4. **`scripts/seed_database.py` の修正**
   - `seed_service_data()` 内に `SERVICE_FEATURES` の投入処理を追加する
   - 新規関数 `seed_sample_tenant_service_features()` を追加し、`SAMPLE_TENANT_SERVICE_FEATURES` を `tenant_services` コンテナに投入する
   - メインの実行フローに上記関数の呼び出しを追加する

## 備考

- `SERVICES` リスト内のドキュメントには既に `"partitionKey": "<id値>"` が含まれている（既存コード確認済み）。パーティションキーの物理定義変更のみが必要。
- DevContainer 再起動で `setup_database.sh` が自動実行され、DB が初期化される。開発中にコンテナ再作成が必要な場合は手動で `bash scripts/setup_database.sh` を実行する。
- `tenant_services` コンテナのパーティションキーは `/tenantId` のまま変更なし。
