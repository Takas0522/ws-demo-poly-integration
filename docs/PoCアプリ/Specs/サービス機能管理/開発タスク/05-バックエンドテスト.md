# 05 — バックエンドテスト

## 概要

サービス機能管理のバックエンド実装に対するユニットテストおよび統合テストを作成する。サービス層のビジネスロジック検証とAPIエンドポイントの結合検証を行う。

## 背景・目的

検証計画（06-検証計画.md）で定義されたテストケース TC-U-01 〜 TC-U-10、TC-I-01 〜 TC-I-10 を実装し、機能の正確性を担保する。既存テスト（`tests/integration/test_service_api.py`）の退行テストも確認する。

## 対象リポジトリ

- `src/service-setting-service/` — テストファイルの新規作成

## タスク詳細

### 実装内容

1. **`src/service-setting-service/tests/test_service_feature_service.py` の新規作成（ユニットテスト）**

   リポジトリをモックし、`ServiceFeatureService` のビジネスロジックをテストする:

   - TC-U-01: `get_service_features` — 正常系（3件返却）
   - TC-U-02: `get_service_features` — サービス存在しない場合（404）
   - TC-U-03: `get_tenant_service_features` — テナント設定ありのマージ確認（`is_default=False` のケースと `is_default=True` のケースが混在）
   - TC-U-04: `get_tenant_service_features` — テナント設定なし、全デフォルト（全 `is_default=True`）
   - TC-U-05: `get_tenant_service_features` — サービス未割り当て（403）
   - TC-U-06: `update_tenant_service_feature` — 新規 upsert 正常系
   - TC-U-07: `update_tenant_service_feature` — 既存 upsert 正常系
   - TC-U-08: `update_tenant_service_feature` — 存在しない feature_id（404）

2. **`src/service-setting-service/tests/test_service_feature_repository.py` の新規作成（ユニットテスト）**

   CosmosDB クライアントをモックし、リポジトリのクエリ構造をテストする:

   - TC-U-09: `get_features_by_service_id` — 正しいクエリとパーティションキーが使用されることを確認
   - TC-U-10: `upsert_tenant_feature` — id 生成が `{tenant_id}_{feature_id}` であることを確認

3. **`src/service-setting-service/tests/integration/test_service_feature_api.py` の新規作成（統合テスト）**

   FastAPI `TestClient` を使用し、エンドポイントの結合テストを行う:

   - TC-I-01: `GET /services/{service_id}/features` 正常系 — 200、レスポンス構造確認
   - TC-I-02: `GET /services/{service_id}/features` 未認証 — 401
   - TC-I-03: `GET /services/{service_id}/features` 存在しないサービス — 404
   - TC-I-04: `GET /tenants/{tenant_id}/services/{service_id}/features` 正常系 — 200、マージ結果確認
   - TC-I-05: `GET /tenants/{tenant_id}/services/{service_id}/features` サービス未割り当て — 403
   - TC-I-06: `PUT` 正常系 — 200、`is_enabled` と `updated_by` の確認
   - TC-I-07: `PUT` 権限不足 — 403
   - TC-I-08: `PUT` サービス未割り当て — 403
   - TC-I-09: `PUT` 存在しない feature_id — 404
   - TC-I-10: `PUT` バリデーションエラー — 422

4. **既存テストの退行確認**
   - `pytest tests/integration/test_service_api.py -v` が既存テスト全件パスすることを確認

## 備考

- ユニットテストでは `unittest.mock.AsyncMock` を使用してリポジトリ/CosmosDB をモックする
- 統合テストは CosmosDB Emulator が起動している環境でのみ実行可能。`@pytest.mark.integration` マーカーを付与する
- 既存の `conftest.py` の `test_client` / `auth_headers` フィクスチャを活用する
- テスト用の JWT トークン生成には、既存のテストパターン（モックした `get_current_user` dependency）を踏襲する
